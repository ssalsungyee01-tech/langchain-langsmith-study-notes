# 자료를 만들며 깨달은 점

이 자료를 만들면서 가장 크게 깨달은 것은, 초보자용 자료는 내용을 쉽게 쓰는 것만으로는 충분하지 않다는 점입니다. 읽는 사람이 실제로 어떤 순간에 멈추는지, 어떤 단어에서 "이게 뭔데?"라고 느끼는지까지 생각해야 합니다.

처음에는 8주 과정처럼 순서를 세워 설명하는 방식이 그럴듯해 보였습니다. 하지만 수업을 듣는 학생 입장에서는 전체 과정을 다시 정독하고 싶은 경우보다, 특정 개념이 헷갈려서 필요한 부분만 찾아보고 싶은 경우가 많습니다. 그래서 하나의 긴 글보다 주제별 게시글 묶음이 더 적합했습니다.

또 하나의 깨달음은 "쉬운 비유"와 "정확한 개념"을 따로 나누는 방식이 항상 좋은 것은 아니라는 점입니다. 구조는 깔끔해 보이지만, 읽는 사람에게는 단어장처럼 느껴질 수 있습니다. 초보자에게 필요한 것은 표제어를 외우는 느낌이 아니라, 앞 문장에서 다음 문장으로 자연스럽게 따라갈 수 있는 흐름입니다.

LangChain과 LangSmith를 가르칠 때 특히 조심해야 할 부분도 분명해졌습니다. 도구, API, 모델명, import 경로는 계속 바뀝니다. 여기에 너무 집착하면 학생은 "버전이 바뀌면 다 무너지는 지식"을 배우게 됩니다. 반대로 메시지, 프롬프트, 도구, RAG, 메모리, 구조화 출력, trace, evaluation 같은 뼈대 개념은 오래 갑니다. 바뀌는 것과 오래 가는 것을 분리해주는 일이 중요합니다.

기초가 약한 학생에게는 곁다리처럼 보이는 개념도 사실은 다리 역할을 합니다. 클래스, 객체, 스키마, DB, DBMS 같은 말은 LangChain의 본론은 아니지만, 모르면 본론에 들어가기 전에 계속 걸립니다. 이런 개념은 깊게 파고들 필요는 없지만, "일단 이렇게 이해하면 된다"는 얕은 발판을 만들어줘야 합니다.

그리고 RAG나 agent 같은 멋진 개념보다 더 중요한 것은 안전과 검증입니다. 모델이 도구를 쓰게 만들 수 있다는 말은 곧 모델이 위험한 행동을 하게 만들 수도 있다는 뜻입니다. 실제 발송, 삭제, 결제, 권한 변경처럼 되돌리기 어려운 작업은 사람의 승인 없이 실행되면 안 됩니다. 외부 문서는 근거가 될 수 있지만 명령이 되어서는 안 됩니다.

마지막으로, LangSmith의 의미도 더 선명해졌습니다. LangSmith는 단순히 실행 기록을 예쁘게 보여주는 도구가 아닙니다. "왜 틀렸는지"와 "정말 좋아졌는지"를 확인하게 해주는 장치입니다. 초보자에게도 trace와 evaluation을 일찍 소개해야 하는 이유가 여기에 있습니다. AI 앱은 답을 만드는 것에서 끝나는 것이 아니라, 과정을 보고 기준으로 개선하는 데서 완성됩니다.

정리하면 이 자료의 방향은 다음과 같습니다.

- 학생이 필요한 구간을 골라 볼 수 있게 주제별 페이지로 나눈다.
- 도구 사용법보다 오래 가는 개념을 먼저 설명한다.
- 초보자가 멈출 만한 단어에는 `이게 뭔데?` 블록으로 짧은 발판을 둔다.
- 모델이 할 일, 도구가 할 일, 사람이 승인할 일을 구분한다.
- LangSmith를 디버깅과 평가의 관점에서 자연스럽게 연결한다.

이 자료가 잘 작동한다면, 학생은 코드를 그대로 외우지 않아도 "LLM 앱이 어떤 흐름으로 만들어지는지"를 자기 말로 설명할 수 있어야 합니다.
