# Structured Output: 앱이 읽을 수 있는 답을 받는 법

모델은 자연어를 잘 만듭니다. 하지만 앱 입장에서는 자연어만으로는 곤란한 경우가 많습니다. 예를 들어 업무 자동화 앱이 이런 답을 받았다고 해봅시다.

> 네, 메일 초안을 작성했습니다. 발송 전 확인이 필요합니다.

사람은 이해할 수 있지만 앱은 이 문장에서 요청 종류, 사용한 도구, 승인 필요 여부, 경고 메시지를 안정적으로 꺼내기 어렵습니다. 그래서 구조화 출력이 필요합니다.

구조화 출력은 모델에게 정해진 데이터 모양으로 답하게 하는 방식입니다.

```json
{
  "request_type": "email_draft",
  "answer": "메일 초안 내용",
  "tools_used": ["draft_email"],
  "sources": [],
  "requires_approval": true,
  "warning": "실제 발송 전 사용자의 확인이 필요합니다."
}
```

여기서 schema라는 말이 다시 나옵니다. schema는 데이터 양식표입니다. 어떤 필드가 있어야 하는지, 각 필드는 문자열인지 숫자인지 참/거짓인지 정해둡니다. 구현 방식은 Pydantic을 쓸 수도 있고, provider가 제공하는 structured output 기능을 쓸 수도 있습니다. 이 부분은 바뀔 수 있습니다.

> #### 이게 뭔데? JSON
> JSON은 데이터를 주고받을 때 많이 쓰는 텍스트 형식입니다. `{ "name": "민지", "age": 20 }`처럼 이름과 값을 짝지어 표현합니다. 사람도 읽을 수 있고 앱도 읽기 좋아서 API 응답이나 AI 출력 형식에 자주 쓰입니다.

> #### 이게 뭔데? Pydantic
> Python에서 데이터 모양을 정의하고 검증할 때 많이 쓰는 라이브러리입니다. LangChain structured output 예시에서 자주 등장합니다. 초보 단계에서는 "출력 양식 검사를 도와주는 도구" 정도로 보면 됩니다.

> #### 이게 뭔데? Validation
> validation은 값이 약속한 형식에 맞는지 확인하는 과정입니다. 예를 들어 `requires_approval`은 참/거짓이어야 하는데 "아마도요" 같은 문자열이 오면 앱이 곤란해집니다. validation은 이런 문제를 잡아줍니다.

구조화 출력은 도구 호출, RAG, 평가와도 연결됩니다. 도구를 어떤 인자로 호출했는지, 어떤 문서를 근거로 썼는지, 승인이 필요한지 같은 정보를 정해진 필드로 남겨야 나중에 trace와 evaluation에서도 확인하기 쉽습니다.

하지만 구조화 출력도 만능은 아닙니다. 모델이 항상 완벽한 JSON을 내보낸다고 믿기보다, 스키마 검증과 오류 처리까지 생각해야 합니다. 오래 가는 원리는 간단합니다. 앱이 후속 처리를 해야 한다면, 모델 답변도 앱이 읽기 쉬운 구조로 받아야 합니다.
